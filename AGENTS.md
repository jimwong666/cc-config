# 开发编码相关偏好
## 编码风格
- 命名偏好
	- js 等代码变量命名，建议以驼峰为主。
		- 也可以使用 "_" 下划线为辅。
		- 局部变量以下划线 "_" 开头。
	- css 等代码变量命名，建议用短横线 "-" 分割。
- 缩进风格
	- 用 tab 缩进。
- 注释
	- 代码中需要添加注释的地方，建议添加注释说明其作用。
	- 注释包括时间、功能说明、入参、返回值等。
	- 注释不宜过长，语言精简、意思明确即可。

## 开发工具与环境约束
- 前端包管理器
	- 默认使用 npm 。
- 版本管理
	- 不要自动推送代码到远程。

## 思维方式
- **第一性原理思考**：不接受任何任务表面的样子，先回到它要解决的根本问题。
	- **深挖根本问题**：先问“这到底再解决说明”，而非“怎么做这个功能”。
	- **质疑前提**：前提存疑就先纠正前提，在执行；别在错的地基上往下做。
	- **拒绝过度设计**：每个动作都要问“是否真的服务根本目标”，警惕为做而做、警惕与现有功能重叠的方案。
	- **收口到最小必要**：方案膨胀时主动收回到这一轮真正必要的范围，有更短路径主动说明，但未经确认时不要擅自执行。
- **不确定就问，别猜**
- **反复受阻就升维**，同一个问题连续三次仍未解决，停止再原思路里面重试 —— 跳出执行和检查的细节，从更系统、更全局的层面重新审视：是不是前提错了？是不是问题定义错了？是不是该换条路径？

## 执行纪律
- **最小化改动**：没要求的不写，只改要求的部分。如果认为其他地方也需要改，说出来，等确认。
- **探索前先报告**：调查或阅读代码库时，先列出打算读的范围，等确认后再行动。
- **简洁优先**：能删代码就不加代码。找根因，不打补丁。

## 完成后输出
- 改了什么。
- 没改什么（如有必要说明原因）。
- 如何验证结果（命令/文件位置/需要手动检查的点）。
- 如果有需要我确认的点，请列出待确认点或者文件详细的位置。如果涉及代码文件，除了提供完整路径外还需要指出具体的行数和内容。

## 持续改进
